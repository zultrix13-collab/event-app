# Feature хөгжүүлэх дараалал (мэдлэгийн зааварчилгаа)

Энэ файл нь **нэг нэгээр** (эсвэл бага багцаар) feature босгох дарааллыг тодорхойлно. Баримтын бүх ID, тайлбар, PoC-ийн шаардлага нь **`docs/SYSTEM_REQUIREMENTS.md`**-д **гол мэдлэгийн сан** болон хадгалагдсан. Нарийвчилсан спринт даалгавар, кодын задрал — **`docs/IMPLEMENTATION_PLAN.md`**. Одоогийн код ба зай — **`docs/GAP_ANALYSIS.md`**.

---

## 1. Одоогийн бүтэц (богино)

| Давхарга | Технологи | Тэмдэглэл |
|----------|-----------|-----------|
| Вэб + Admin | Next.js, Supabase | Баримтад заасан Docker/K8s/микросервис бүрэн биш — шилжилт төлөвлөх |
| Апп | Flutter, Supabase | Вэбтэй ижил `profiles`, хөтөлбөр, wallet UI-ийн суурь зэрэг бий |
| Төлбөр | QPay (billing), wallet төлөвлөгөө | PCI-DSS, tokenization, idempotency — баримтын заавал шаардлага |

Баримт (NDC MCloud, API Gateway, Kafka/RabbitMQ, Elasticsearch гэх мэт)-тай бүрэн нийцэх нь **тусад инфра/архитектур шат** шаардна.

---

## 2. Эхний 2 долоо хоног — заавал PoC (5 чиглэл)

Баримтын **6-р хэсэг**. Аль нэг буруу бол бүтээгдэхүүний гол урсгал эвдэгдэнэ. Эдгээрийг **параллель** (өөр баг) эсвэл дараалсан ч болно; хамгийн багадаа **PROG-02** + **PAY-01**-ийг эхлүүлэх нь бусад модулийг түгждэггүй.

| Дараалал | ID | Юу баталгаажуулах вэ |
|----------|-----|------------------------|
| 1 | **PROG-02** | Суудлын бүртгэл — DB түвшний lock/RPC, олон зэрэг хүсэлт (жишээ нь 500) — давхар бүртгэл 0 |
| 2 | **PAY-01** (+ idempotency) | Wallet гүйлгээний ACID, `idempotency_key`, давхар цэнэглэлт/хасалтгүй |
| 3 | **AUTH-03** | Offline HMAC шалгалт, dynamic QR, iOS/Android NFC (NDEF) — хугацаа дууссан/засварласан payload татгалзах |
| 4 | **AI-01, AI-02** | Монгол embedding + retrieval, 20 асуултын benchmark, эх сурвалж заах (hallucination бууруулах) |
| 5 | **MAP-04** (MVP) | Beacon-гүй эхлээд: **QR checkpoint + SVG давхарга** дээр байршил — tender-д beacon тусдаа ажил гэж заасан |

**PAY-03** (QPay/SocialPay sandbox) болон webhook retry-г PoC эсвэл шууд Sprint 2-т холбоно (`IMPLEMENTATION_PLAN`).

## 3. Үндсэн хөгжүүлэлт — feature-ээр дараалал (нэг нэгээр барих зөвлөмж)

Доорх нь **хамаарал** (dependency) ба **заавал** шаардлагыг урьдчилсан дараалал. Алхам бүрийн дэлгэрэнгүй даалгавар `IMPLEMENTATION_PLAN.md`-д байна.

### Фаз A — Аюулгүй байдал ба удирдлагын суурь

1. **Audit log** өргөтгөл (баримт 4.3) — операторын үйлдэл, чухал API (`operator_audit` зэрэгтэй уялдуулах).
2. **2FA** — Admin, Мэргэжилтэн (`SUPABASE` MFA эсвэл TOTP — middleware-д `aal2` шаардлага аль хэдийн заагдсан).
3. **Admin IP хязгаарлалт** — reverse proxy / edge дээр whitelist, dev-д feature flag.
4. **AUTH-05** RBAC нягтруулалт — вэб + API endpoint зэрэгцээ.
5. **AUTH-02** (Flutter) — `local_auth`, fallback PIN/нууц үг.

### Фаз B — Нэвтрэлт ба хэрэглэгчийн урсгал

6. **AUTH-01** — OTP урсгал, rate limit (одоогийн RPC-тай нийцүүлэх).
7. **AUTH-04** SSO/JWT — олон клиент нэг сессийн стратеги (урт хугацааны ажил бол баримтаас зэрэглэл бууруулж болно).
8. VIP урсгал — **pending approval**, `apply-vip`, admin баталгаажуулалт (баримт 2-р хэсэгтэй нийцнэ).

### Фаз C — Хөтөлбөр (ядрын бүтээгдэхүүн)

9. **PROG-02** — PoC-оос production (индекс, ачааллын тест).
10. **PROG-01** — календарь, шүүлт.
11. **PROG-04** — QR/NFC ирц (AUTH-03-тай уялдах).
12. **PROG-03** — My Agenda, iCal/export.
13. **PROG-06** — илтгэгч admin (бодит цагийн шинэчлэлт).
14. **PROG-05** — санал асуулга.

### Фаз D — Төлбөр (эрсдэл өндөр — дараалал чухал)

15. **PAY-01** — wallet бүрэн (DB + сервер action + апп).
16. **PAY-03** — QPay + SocialPay, webhook + idempotency.
17. **PAY-05** — нэгдсэн орлогын тайлан (admin dashboard).
18. **PAY-02** — карт tokenization (PCI — гуравдагч PSP/tokenization лүү түлхэх).
19. **PAY-04** — дэлгүүр/IAP — StoreKit 2 / Play Billing, бодлогын зөрчилгүй загвар.

### Фаз E — Үйлчилгээ

20. **SVC-07** Lost & Found (заавал, бага төвөгтэй — эрт авах нь зөв).
21. **SVC-08** үйлчилгээ үзүүлэгчийн бүртгэл + admin KYC workflow.
22. **SVC-01** дэлгүүр — сагс, checkout → wallet/PAY.
23. **SVC-04, SVC-05, SVC-03** — ресторан, буудал, нисэх буудлын тээвэр (гэрээ/API бэлэн болохоор).
24. **SVC-02** тээвэр — гуравдагч API + тасалдлын fallback.
25. **SVC-06** e-SIM — операторын гэрээ шаардлагатай тул сүүлд эсвэл тусад нь төсөл.

### Фаз F — Газрын зураг

26. **MAP-05** offline татах + кэш (апп).
27. **MAP-01** гадаад газрын зураг (Mapbox/Google — лиценз).
28. **MAP-02** дотоод төлөвлөлт (байршлын өгөгдөл).
29. **MAP-03** навигаци.
30. **MAP-04** — QR MVP-аас **BLE hybrid** руу өргөтгөл (beacon суурилуулалттай уялдуулах).

### Фаз G — Мэдэгдэл, тусламж, ногоон модуль

31. **NOTIF-01, NOTIF-04, NOTIF-06** — push, онцгой байдал, шүүлт (FCM/APNs — `IMPLEMENTATION_PLAN` 1.4).
32. **NOTIF-03** имэйл, **NOTIF-02** SMS (оператор).
33. **NOTIF-05** орчуулга (заавал биш).
34. **Help / emergency** — баримт 10-р хэсэгт заасан найдвартай урсгал (дуудлага, журам).
35. **GREEN-04** — арга хэмжаа дуусахад алхмын өгөгдөл устгах (**заавал**, бага төвөгтэй).
36. **GREEN-01, GREEN-02** — HealthKit / Health Connect, CO₂ боловсролын тооцоо.
37. **GREEN-03** — leaderboard, badge (**дараа нь**).

### Фаз H — AI, CMS, вэб контент

38. **Headless CMS** (Strapi/Sanity/Contentful) — олон хэл, workflow Author→Editor→Publisher, хуваарьт нийтлэх.
39. **AI-01–AI-05** — RAG production, human handoff, ярианы log + GDPR.
40. **Вэб хэсгүүд** (баримт 9) — Blue/Green zone, Elasticsearch хайлт, header widget (цаг агаар, CO₂, chatbot).

### Фаз I — Инфра, нийцлэл, гарц

41. **Staging** = production-той ижил тохиргоо; **blue-green**; **health/readiness** probe.
42. **Ачааллын тест** — 10k concurrent зорилт (`SYSTEM_REQUIREMENTS` 4.2).
43. **Penetration test** — гадаад аудит (баримт 4.3).
44. **ISO 27001 / GDPR** баримт бичиг, нөөцлөлт, шифрлэлийн шалгалт (инженерийн хийсвэр + процесс).

## 4. Вэб vs апп — хэн юу хийх вэ (хамраалал)

| Хүрээ | Вэб (Next.js) | Flutter |
|--------|----------------|---------|
| Зохион байгуулагч, төлбөр, super admin | Гол | Хязгаарлагдмал |
| Оролцогч: хөтөлбөр, газрын зураг, үйлчилгээ | `/app/*` | Tab-ууд (баримт 10) |
| Дижитал ID / ирц | Сервер талын signing, admin | QR/NFC харуулах, offline verify |
| Төлбөр | Webhook, admin тайлан | Wallet, QR scan төлбөр |

---

## 5. Энэ файлыг хэрхэн ашиглах вэ

- Шинэ feature эхлүүлэхэд: **энэ дарааллын дугаар** → `SYSTEM_REQUIREMENTS.md`-аас ID-г нээн шалгах → `IMPLEMENTATION_PLAN.md`-аас алхмуудыг хуулах.
- Баримт шинэчлэгдвэл эхлээд **`SYSTEM_REQUIREMENTS.md`**-ийг шинэчилж, дараа нь энэ дарааллыг тохируулна.
